Proposed System Analysis

Smart School Bell and Announcement Automation System for FCU Elementary Department

Overall Evaluation

Among the proposals reviewed so far, this is one of the more feasible hardware-software capstone projects. The problem is specific, the users are clearly identified, and the system has a direct operational purpose.

Unlike many projects that attempt to solve broad societal issues, this proposal focuses on a clearly defined institutional process that can realistically be automated.

However, the project still has weaknesses in problem justification, novelty, and scope definition.

Strengths

1. Real and Existing Problem

The problem exists within the FCU Elementary Department.

Current situation:

  • Bell operation depends on a person.
  • Human delays can affect schedule consistency.
  • There is no automated scheduling mechanism.

This is a legitimate operational problem.

2. Clear Users

Beneficiaries are identifiable:

  • Elementary Department
  • School administrators
  • Teachers
  • Students

This makes testing and evaluation easier.

3. Strong BSIT Components

The project includes:

  • Embedded systems
  • Database
  • Web application
  • Scheduling engine
  • Hardware integration
  • Audio automation

These are appropriate BSIT competencies.

4. Feasible Implementation

Compared to:

  • AI systems
  • Marketplace platforms
  • Mobile games

this project is more achievable within the capstone timeline.

Weaknesses

1. Problem Severity is Overstated

Current justification:

"Teachers and students don't know when it's time for break or next subject."

This may be challenged by a panel.

Realistically:

  • teachers already have schedules
  • students know approximate class times
  • occasional late bell ringing does not create complete confusion

The problem should focus on:

lack of automation and scheduling consistency

instead of implying severe operational disruption.

2. Novelty is Weak

A school bell timer already exists.

Commercial products already provide:

  • automated bells
  • programmable schedules
  • announcement systems

Therefore:

Weak novelty claim:

"We automated the bell."

Stronger novelty claim:

"A localized web-managed school scheduling and emergency notification platform integrated with programmable bell automation."

The novelty is in the integration, not the bell itself.

3. Push-to-Talk Paging May Be Unnecessary

Current features include:

  • Push-to-Talk Paging
  • Custom Audio Alerts
  • Bell Scheduling

These are essentially separate systems.

The danger is scope expansion.

"Is this a bell system or a school-wide communication system?"

4. Emergency Use Case Missing

This is actually a missed opportunity.

The system would become more valuable if it included:

  • emergency alarm activation
  • evacuation signal
  • lockdown signal
  • urgent school announcements

This provides stronger justification than merely ringing bells.

Scope Assessment

Must Keep

  • Automated Bell Scheduling
  • RTC Integration
  • Web-Based Schedule Management
  • Manual Override
  • Audio Output

These are enough for a complete capstone.

Optional Features

  • Emergency Alarm Mode
  • Special Event Scheduling
  • Different Bell Patterns

Consider Removing

  • Push-to-Talk Paging
  • Complex Audio Decoder Features

Unless these are already proven feasible.

Current Progress Assessment

Positive

Already completed:

  • Time Management Engine
  • Web Dashboard Interface

This is a good sign.

Concern

Remaining work includes:

  • hardware integration
  • audio modules
  • enclosure fabrication

For hardware projects, these are usually the most difficult parts.

They should prioritize hardware immediately.

Recommended Revised Problem Statements

  1. The FCU Elementary Department relies on manual bell operation to signal class schedules and daily activities.
  2. There is no automated scheduling mechanism to ensure accurate and consistent bell activation.
  3. Administrators have no centralized platform for managing and modifying bell schedules.
  4. The current system does not provide automated emergency alerts and announcements when needed.
  5. There is no digital logging and monitoring system for scheduled bell and announcement activities.

Recommended Objectives

General Objective

To design and develop a Smart School Bell and Announcement Automation System that improves schedule management and school-wide notification processes in the FCU Elementary Department.

Specific Objectives

  1. To develop an automated bell scheduling system using a Real-Time Clock (RTC) module.
  2. To create a web-based platform for managing and updating bell schedules.
  3. To implement customizable audio alerts and announcements.
  4. To provide manual override and emergency alarm functionality for authorized personnel.
  5. To develop a database system for storing schedules and activity logs.

Novelty Rating

Originality: 6/10

Technical Complexity: 8/10

Practical Value: 9/10

Implementation Feasibility: 8/10

Coordinator Verdict

Status: Approved with Minor Revision


Reasons

  • Real and localized problem.
  • Strong hardware-software integration.
  • Clear beneficiaries.
  • Good implementation progress compared to most groups.
  • Scope is generally manageable.

Recommendations

  • Reframe the problem around automation and emergency communication rather than occasional secretary delays.
  • Strengthen novelty by emphasizing integrated scheduling, monitoring, and emergency notification features.
  • Reduce non-essential communication features if hardware integration becomes difficult.
  • Prioritize hardware and audio module development immediately because these are now the highest-risk components.

Of the proposals reviewed so far, this is one of the stronger candidates for successful completion and deployment, provided the group maintains a controlled scope and focuses on completing the hardware integration early.
