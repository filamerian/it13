Key Remaining Actions

  1. 1) Add localized quantitative data for Roxas City

    Why: Quantitative local data turns claims into evidence — strengthens problem statements, objectives, and significance.
    How (steps):

    • Request data from LGU/Transport Office: monthly complaints, route ridership, traffic/incident logs, PUVMP local reports.

    • If LGU data is unavailable, do field measurements: select 4–6 key stops, record bus arrivals over several days (peak vs off-peak) and compute headways, waiting times, and passenger counts.

    • Run a short commuter survey (n≥100 ideal; n≥30 minimal) asking wait times, missed trips, satisfaction.
      Metrics to include: average waiting time (min), headway (min), % of commuters reporting >15min wait, monthly complaints, ridership per route.
      Example table:
      | Route | Avg wait (min) | Headway (min) | Ridership/day | Complaints/mo |
      |---|---:|---:|---:|---:|
      | Green | 12.4 | 18 | 1,200 | 8 |
      | Red | 15.6 | 22 | 900 | 12 |
      Where to insert: Overview (Chapter I) and Appendix (raw data).
      Deliverable: 1 small stats table + 1 short paragraph interpreting results; optional chart (bar or line).

    2) Provide a modular feature breakdown + mockups

    Why: A modular breakdown clarifies scope, responsibilities, and development milestones. Mockups communicate UX expectations to evaluators.
    How (steps):

    • Define modules (suggested list below). For each, list features, inputs/outputs, and acceptance criteria.

    • Create low-fidelity wireframes (hand sketches or Figma/draw.io). Include 3 screens: main map (live buses), ETA detail/stop view, operator/dispatcher dashboard.
      Suggested modules:

    • Tracking Module: GPS ingestion, device management, heartbeat.

    • UI Module: mobile/web mapping, stop lookup, notifications.

    • Analytics Module: historical logs, utilization, KPI dashboards.

    • Operator Module: dispatch console, rerouting, alerts.

    • Signage Module: terminal display rendering.

    • Security & Privacy Module: auth, encryption, PII controls.
      Example feature entry:
      Tracking Module — ETA accuracy ≤ +/- 60s; updates every 10s; supports up to 50 devices per route.
      Where to insert: Desired State (Chapter I) and System Design (Chapter III) with mockups as figures or appendix.
      Deliverable: Modular table + 3 wireframe images.

    3) Include expansion plans for other PUV types (phased rollout)

    Why: Shows realistic scalability and policy awareness (jeepneys/tricycles differ technically and organizationally).
    How (steps):

    • Define phases: Pilot → Scale to similar buses → Retrofit jeepneys/multicabs → City-wide integration.

    • For each phase list prerequisites (hardware, funding, stakeholder agreements), estimated timeline, and risks.
      Example phases:

    • Phase 1 (0–3 months): Green & Red buses (pilot).

    • Phase 2 (3–9 months): Add remaining modernized PUVs; terminal signage rollout.

    • Phase 3 (9–18 months): Retrofit select jeepneys/multicabs; API for third-party apps.
      Where to insert: Desired State (end of section) and Scope & Limitations (note planned future scope).
      Deliverable: 1 short phased plan table.

    4) Add measurable targets in objectives (SMART objectives)

    Why: Measurable targets let evaluators assess success objectively.
    How (steps):

    • Convert each specific objective to a SMART statement (Specific, Measurable, Achievable, Relevant, Time-bound).
      Suggested targets:

    • “Reduce average commuter waiting time by 30% on pilot routes within 6 months of deployment.”

    • “Achieve ETA prediction accuracy ≥ 80% within ±1 min during field tests.”

    • “Reach user satisfaction of ≥ 85% (post-use survey) among pilot users (n≥40) after 3 months.”

    • “Maintain system uptime ≥ 95% during pilot.”
      Where to insert: Objectives of the Study (Chapter I) and Evaluation Metrics (Chapter III).
      Deliverable: Revised objectives list with exact numeric targets and measurement method.

    5) Clarify theory distinctions (avoid overlap)

    Why: Distinct theory roles strengthen theoretical grounding and show thoughtful alignment between theory and design.
    How (steps):

    • For each theory, write 1–2 sentences tying it to a single, unique aspect of the system. Avoid repeating the same application across theories.
      Suggested mapping (one short line each):

    • Diffusion of Innovations — adoption strategy (pilot, communication, observability).

    • Systems Theory — describes system boundaries, data flows, and environmental interactions (hardware ↔ network ↔ users).

    • Task-Technology Fit — ensures features directly support commuter tasks (trip planning, checking ETA).

    • Stakeholder Theory — prioritizes requirements among LGU, operators, commuters.

    • HCI — informs interaction design, accessibility, and usability testing.
      Where to insert: Theoretical Framework — replace overlapping paragraphs with succinct mapping table.
      Deliverable: Revised table and 1 paragraph explanation per theory.
