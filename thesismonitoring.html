Capstone Proposal Evaluation

Design and Implementation of a Thesis Monitoring and Contribution Tracking System for FCU

Final Verdict: REJECT (Not Recommended for Capstone Implementation)


This proposal is not suitable for approval in its current concept and should be reframed or replaced entirely.

Core Reason for Rejection

The fundamental issue is not technical, but conceptual:

The system attempts to measure “student contribution” and “inactivity,” which is not a valid or reliable function of an information system.

A BSIT capstone must solve data, process, or system inefficiency problems, not attempt to judge human effort or fairness in group behavior.

Major Evaluation Points

1. Problem is Misclassified as an IT Problem

The stated issue is:

“Inactive or non-contributing group members in thesis groups.”

This is primarily:

  • behavioral
  • academic
  • interpersonal
  • adviser-managed

not a system automation or data management problem.

A system cannot objectively determine “contribution” without bias or assumptions.

2. “Contribution Tracking” is Not Technically Defensible

The system assumes:

  • activity = contribution
  • logs = effort
  • reports = performance

This is flawed.

A student can:

  • log activity without meaningful output
  • contribute significantly without digital traces
  • be unfairly flagged due to lack of system interaction
This creates false accountability, which is academically risky.

3. High Risk of Ethical and Academic Issues

The system introduces potential issues such as:

  • student conflict escalation
  • biased reporting between members
  • misuse as complaint or accusation platform
  • unfair labeling of “inactive” students

This shifts the system from monitoring tool to judgment system, which is not appropriate for a capstone.

4. Adviser Role is Already the Proper Solution Channel

Current academic structure already handles this through:

  • adviser consultations
  • defense evaluations
  • group grading mechanisms
  • oral defenses and documentation review

The system duplicates a process that already exists and is better handled manually by academic supervisors.

5. Weak Technical Novelty

The system mainly includes:

  • login system
  • reporting module
  • progress tracking
  • notifications

These are:

  • standard CRUD features
  • common in many existing systems
  • not research-level innovations

There is no clear technical advancement or unique computing problem being solved.

6. Problem Statement is Based on Assumption, Not Evidence

The proposal claims:

“Students experience unfair workload and inactive members.”

However:

  • no survey data is presented
  • no adviser validation is provided
  • no measurable baseline exists

This makes the problem anecdotal rather than research-based, which is unacceptable for capstone justification.

Why This Cannot Be Approved as a Capstone

A BSIT capstone must demonstrate:

  • a clear information system problem
  • measurable system improvement
  • objective data handling
  • technical contribution

This proposal instead focuses on:

evaluating human behavior and assigning accountability

which is outside the scope of reliable system design.

Final Recommendation

* Do NOT proceed with this system as proposed


* Do NOT implement “contribution tracking” or “inactive member detection”

Suggested Direction (If They Want to Save the Idea)

Thesis Project Management System

Focus only on:

  • project milestones
  • document submission tracking
  • adviser feedback system
  • meeting scheduling
  • file repository
  • progress dashboards

Remove:

  • reporting of “inactive members”
  • contribution scoring
  • behavioral evaluation

Closing Assessment

This proposal, in its current form, is not aligned with BSIT capstone standards, introduces unverifiable behavioral measurement, and risks creating academic and ethical complications.


It is therefore not recommended for approval and should be redesigned from the ground up before reconsideration.
